# 02_Reduction：从分支发散到线程粗化，逼近带宽极限的艺术

## 1. 为什么需要归约（Reduction）机制？
当面临求解全局和（Sum）、最大值（Max）、或者算子底层的均值方差时，不能像 Element-wise 那样让各个线程互不干扰。这就引出了以**树状结构（Tree-based algorithm）**进行并行合并的核心模式。

## 2. 演进路线剖析

### 第一阶段：消除 Warp Divergence
如果在代码中使用 `if (tid % (2 * stride) == 0)`，会使得同一个 Warp（32个线程组成的执行束）中的部分线程闲置，这直接违背了 SIMT（单指令多线程）的设计哲学。
**优化手段**：通过改变索引映射公式计算（跨步访问），让活跃的线程永远集中在 Warp 的前半部分，直到线程数降低。

### 第二阶段：Shared Memory 加速
与 `01_Basics` 的矩阵乘法相似，归约每缩减一次都需要覆盖写入和读取。如果数据在全局内存中，将产生巨大的访问延迟。
**优化手段**：让每个线程将所属元素拉到 Shared Memory 并通过 `__syncthreads()` 同步，之后的树状相加直接在 L1 Cache 级别的 Shmem 中极速完成。

### 第三阶段：线程粗化 (Thread Coarsening) 与原子扩展
一个深刻的反直觉道理是：**更多的线程并不总是更好**。
如果启动极多的线程，有一半的线程在第一轮加法后就闲置了。
**线程粗化**让一个线程先用寄存器连续累加 4 个（或更多）元素。
- 这不仅直接减少了 3/4 的线程同步开销；
- 而且更进一步增加了指令级并行（ILP）。
当我们面对跨越 Block 的千万级数据时，必须利用 `atomicAdd` 让每个 Block 最终归约完的一个标阶汇入全局槽。在这里我们也演练了针对无原生浮点原子实现的 `atomicCAS` （Compare-and-Swap）技巧。

## 3. FMA 点积性能炸裂之路
当我们在 `03_dot_product` 将两组数据先相乘再做归约时，使用 `fmaf(a, b, c)` 指令取代了 `a*b+c`，这不仅用一条指令融合了乘法与加法缩短了管线，还消除了中间浮点舍入降低了误差。
在实测 RTX 4090 上，百万级数据的 FMA 点积利用 L2/L1 Cache 连击跑出 1510 GB/s 的惊人有效带宽，说明我们在瓶颈前彻底压榨干了内存数据馈送管道。
