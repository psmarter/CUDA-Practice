# 03_Scan：打破依赖壁垒，从 Kogge-Stone 到多块分段扫描

## 1. 为什么 Scan 是最难的并行算子之一？
归约（Reduction）只需要吐出一个最终结果，而 Scan（前缀和）需要吐出一个与原本数组相同长度的结果，且每一个元素都依赖于它前面的**所有**元素！
例如 `[1, 2, 3, 4]` 的独占前缀和需要变成 `[0, 1, 3, 6]`。这种严密的依赖网仿佛是串行计算的专属领地。

## 2. 块内算法：KS vs BK 的权衡

在 `prefix_sum.cu` 中，介绍了两种截然不同的数学拓扑。
### 2.1 Kogge-Stone
每次都让距离为 $2^d$ 的远端元素加过来。
- **优点**：代码简单，跑得快，极其适合小规模（如单个 Block）。
- **缺点**：总工作量大达 $O(N \log N)$ 级别。随着数组膨胀，大量的多余加法浪费了硬件的 ALU，功耗急剧上升。

### 2.2 Brent-Kung
受限于 Work-Efficiency 理论，提出了先构建一个完整的二叉树（Up-Sweep），然后向子节点下发累计和（Down-Sweep），两阶段执行。
- **代价**：步骤深度为 $2 \log N$，常数倍的控制流开销。如果在单块内测试 1024 大小，它比 KS 会略慢。但在未来处理数十兆的数据块时，BK 是保证显卡不被海量无用计算淹没的唯一道路。

## 3. 跨越 Block 的鸿沟：递归块级 Scan
如果数据有千万个，由于 CUDA 缺乏跨 Block 的全局强制同步（Global Barrier 原生并不好写），需要：
1. **分治处理**：切成若干个 Block-sized 的小块。对每个块进行独立的（或者结合了 Thread Coarsening 的）局部 Scan。
2. **提取骨架**：把每个块的最后一个汇总值（`block_sum`）拿出来，形成一个小数组。
3. **二次扫描**：因为这个数组通常足够小，用单个 Block 就能对这个 `block_sums` 数组发起一次单独的 Scan。
4. **统一分发**：最后使用第三个 Kernel，让之前保留的骨架扫描值如同基准水位线一样，加上局部元素。

这种 "Divide and Conquer" （分治）算法正是 `02_segmented_scan` 的核心。

## 4. 浮动误差启示
在 1M 数据量的实测中发现，GPU 返回的 `471823.41` 和 CPU 跑出的 `471818.66` 虽然有数值分歧，但这完全合法，这是因为大批量的 `float32` 在不同的结合律（并行树 vs CPU从左到右累加）下被吃掉了不同的尾数。想要解决必须使用 `double` 精度累加，这也给我们日后的混合精度训练打下了一剂基础预防针。
